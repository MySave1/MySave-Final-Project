# 개인화 북마크 서비스 — 리스크 관리 문서 (기획 / PM)

## 1. 문서 목적

이 문서는 **‘개인화 북마크 서비스’ 프로젝트의 기획자(PM)**로서 프로젝트를 진행하며 마주했던 **일정, 리소스, 커뮤니케이션, 기술적 문제들을 어떻게 관리하고 해결했는지** 기록한 것입니다.

특히 시험 기간이라는 제약 속에서

- **초기 기획과 실제 개발 결과물 사이의 차이(Gap)**를 줄여나간 과정
- **팀 내 가교 역할(R&R)**을 수행하며 배운 점

을 중심으로 정리했습니다.

---

## 📌 PM(Role & Responsibility) 역할 정의

이번 프로젝트에서 저는 단순한 기획을 넘어 다음과 같은 매니지먼트 역할을 맡았습니다.

- **일정 및 우선순위 조정**  
  학업과 병행해야 하는 상황을 고려하여 MVP 범위를 설정하고 스프린트 일정을 관리했습니다.

- **기획 문서 현행화**  
  개발 과정에서 변경되는 사항들을 반영하여 기획안(v1, v2…)을 지속적으로 업데이트하고 이력을 관리했습니다.

- **커뮤니케이션 허브**  
  디자이너가 추구하는 사용자 관점과 개발자가 중요시하는 구현 관점 사이의 의견을 조율했습니다.

- **문제 해결 지원**  
  배포나 연동 과정에서 기술적인 이슈가 발생했을 때, 로그 분석 및 문제 해결 과정에 직접 참여했습니다.

---

## 2. 리스크 관리 개요 및 기준

### 2-1. 관리 중점 사항

- **일정 관리**  
  시험 기간과 프로젝트 개발 일정이 겹치는 상황에서 발생할 수 있는 리스크를 최소화하는 데 집중했습니다.

- **요구사항 관리**  
  초기 기획안과 실제 구현 내용이 달라질 경우, 즉시 스펙을 조정하여 혼선을 줄였습니다.

- **팀워크**  
  개발 영역(배포, 연동)에서 이슈가 발생했을 때, PM으로서 적극적으로 지원하며 팀워크를 다졌습니다.

---

### 2-2. 리스크 평가 기준

프로젝트 리스크는 다음 두 가지 기준으로 나누어 평가했습니다.

- **발생 가능성 (Probability)**: 낮음 / 중간 / 높음  
- **영향도 (Impact)**: 낮음 / 중간 / 높음  

**관리 원칙**  
발생 가능성과 영향도가 모두 **높음**으로 평가된 리스크는 **최우선 관리 대상**으로 지정하여 즉각적인 대응 전략을 수립하고 지속적으로 모니터링했습니다.

---

## 3. 리스크 등록부 (Risk Register)

프로젝트 진행 중 실제로 발생했거나 발생 가능성이 높았던 리스크들을 중심으로 정리했습니다.

각 리스크는 **문제 식별 → 대응 전략 → 해결 과정 → 성과**의 흐름을 파악할 수 있도록 구성했습니다.

| 번호 | 구분 | 리스크 내용 | 평가 (P / I) | 대응 전략 및 해결 과정 | 결과 및 개선 효과 |
| --- | --- | --- | --- | --- | --- |
| 1 | 일정 | 시험 기간 중복으로 인한 개발 진척도 저하 | 높음 / 높음 | - MVP 기준 우선순위 재조정<br>- 핵심 기능 외 부가 기능 후순위 조정<br>- 시험일 제외 집중 스프린트 운영 | - 팀원 번아웃 방지<br>- 필수 기능 구현 완료<br>- 개발 지연 리스크 약 40% 감소 |
| 2 | UI/UX | 사용자 편의성을 놓친 초기 디자인 | 중간 / 중간 | - 레퍼런스 기반 페어 디자인 진행<br>- “3초 안에 저장 가능한가” 기준으로 UX 개선 | - 사용자 친화적 UI 도출<br>- 재작업 비용 절감 |
| 3 | 소통 | 기획안(v1)과 개발 화면의 불일치 | 높음 / 높음 | - 기획 문서 현행화(v2, v3)<br>- Wireframe 중간 리뷰 및 즉각 피드백 | - 기획 의도와 결과물 간 차이 해소<br>- 유지보수 효율 증가 |
| 4 | 기술 | 서버 배포 및 프론트–백엔드 연동 오류 | 중간 / 높음 | - 서버 로그 분석 참여<br>- API 경로, CORS 등 연동 포인트 점검 | - 배포 안정화<br>- 트러블슈팅 시간 약 50% 단축 |

## 4. 상세 대응 사례 (Case Study)

### 📌 Case 1. 프론트엔드 화면 불일치 문제

**문제 상황**  
기획서에는 리스트형 UI로 정의했으나, 개발 과정에서 카드형 UI로 구현되는 등 결과물이 달라지는 문제가 발생했습니다.

**PM의 대응**  
기술적 제약을 먼저 파악한 후 기획안을 v2, v3로 업데이트하여 현실적인 구현 상태를 문서에 반영했습니다.  
다만 UX에 치명적인 부분은 근거를 제시하며 수정 방향을 제안했습니다.

**성장 포인트**  
기획안은 고정된 규칙이 아니라, 개발 과정에서 함께 조율해 나가는 **가이드라인**임을 인식하게 되었습니다.

---

### 📌 Case 2. 백엔드 배포 트러블슈팅 참여

**문제 상황**  
EC2 배포 환경 설정 이슈로 프론트엔드와 API 통신이 되지 않는(Network Error) 문제가 지속되었습니다.

**PM의 대응**  
기획자 역할에 국한되지 않고 로그 분석에 참여하여 원인을 파악했습니다.  
Swagger 주소와 실제 프론트 호출 API 주소 불일치를 발견해 문제 해결에 기여했습니다.

**성장 포인트**  
기획자 역시 서버 구조와 API 흐름을 이해하고 있을 때, 팀의 문제 해결 속도를 높일 수 있음을 체감했습니다.

## 5. 결론 및 회고 (Retrospective)

### 💡 잘한 점 (Keep)

- **사람 중심의 일정 관리**  
  시험 기간이라는 특수 상황 속에서도 무리한 일정 강요 없이 선택과 집중을 통해 프로젝트를 완주했습니다.

- **역할 경계를 넘는 협업**  
  기획에만 머무르지 않고 디자인, 배포 과정에도 참여하며 팀의 빈틈을 보완했습니다.

---

### 💡 개선할 점 및 한계 (Limitations)

- **리스크 관리의 한계**  
  사전 예측보다 사후 대응에 의존한 점이 아쉬웠으며, 향후에는 선제적 리스크 식별을 강화할 필요를 느꼈습니다.

- **초기 소통 부족**  
  기획 초반 리뷰를 더 촘촘히 진행했다면 화면 불일치로 인한 비용을 줄일 수 있었을 것이라 판단합니다.

---

## 6. 프로젝트 진행 소감 (Personal Reflection)

이번 프로젝트는 **기획 경험이 전혀 없던 제가 기획자(PM) 역할을 맡아 프로젝트 전반을 총괄하며 성장할 수 있었던 계기**였습니다.

처음에는 기획자의 역할을 명확히 이해하지 못한 채 시작했지만, 프로젝트를 진행하며 기획자는 단순히 화면을 정의하는 사람이 아니라 **팀 전체의 방향과 흐름을 책임지는 역할**임을 깨닫게 되었습니다.

프로젝트 기간 동안 저는 일정, 요구사항, 리스크를 종합적으로 관리하며 매주 진행 상황을 정리해 **주간 발표 형태로 공유**했습니다.  
이를 통해 단순 진척 보고가 아닌, 지연 가능성과 기술적 이슈를 사전에 공유하여 팀원들이 같은 우선순위를 인식하도록 조율했습니다.

또한 디자이너, 프론트엔드, 백엔드 개발자 간 의견 차이가 발생할 때 중재 역할을 수행하며, 기획 의도를 일방적으로 전달하기보다 각 파트의 제약을 이해하고 현실적인 의사결정을 내리고자 했습니다.

프론트–백엔드 연동 오류 및 배포 이슈 발생 시에도 기존 개발 경험을 바탕으로 문제 해결 과정에 직접 참여하며, 기술 이해도가 커뮤니케이션 비용과 리스크 대응 속도에 큰 영향을 미친다는 점을 체감했습니다.

이번 프로젝트를 통해 저는 **프로젝트를 끝까지 완주할 수 있도록 팀을 지원하는 것이 PM의 본질적인 역할**임을 배웠으며,  
기획자로서의 시야와 개발자로서의 이해를 동시에 확장할 수 있었습니다.

앞으로는 이번 경험을 바탕으로 **기술적 이해를 갖춘 기획자이자 선제적으로 리스크를 관리하는 PM**으로 성장해 나가고자 합니다.
