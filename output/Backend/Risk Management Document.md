# 개인화 북마크 서비스 — 리스크 관리 문서 (백엔드)

## 1. 문서 목적
본 문서는 개인화 북마크 서비스 개발 과정에서 발생한 기술적/환경적 문제를 분석하고, 해결 과정과 대응 전략을 정리한 리스크 관리 자료입니다. 개발 및 배포 전 구상 단계에서 예상하지 못했던 형태의 문제들이 다수 발생하였으며, 이를 해결하는 과정에서 시스템 구조 이해, 배포 환경 구성, OAuth 인증 처리, 로그 분석, DB 스키마 관리 등 다양한 기술적 학습 경험을 얻게 되었습니다.

---

## 2. 리스크 정의 기준

### ✔️ 영향도
*프로젝트 서비스 품질, 기능 완성도, 사용자 경험에 영향을 미치는 정도*
* **High:** 서비스 핵심 기능 중단
* **Medium:** 기능 품질 저하 및 사용자 불편 발생
* **Low:** 서비스 운영에 미치는 영향이 낮음

### ✔️ 심각도
*문제가 프로젝트 성과에 미치는 치명도*
* **High:** 장애 발생 시 서비스 불가
* **Medium:** 해결 필요하지만 전체 품질에는 큰 영향 없음
* **Low:** 단순 오류 수준

### ✔️ 우선순위
* 영향도 × 심각도 평가를 통해 대응 순서를 결정

## 4. 교훈 및 개선점 (Lessons Learned)

* **인증 시스템 연동은 사전 API 문서 분석이 필수**
    * OAuth 인증 과정에서 Redirect URI 불일치 문제는 API 흐름을 충분히 이해했다면 예방할 수 있었음.
* **배포 환경 설정은 로컬과 별도로 설계해야 함**
    * 로컬 환경과 prod 서버 환경이 다르므로 별도 설정 관리가 중요함. 특히 DB 연결, 포트, IP 접근 권한 설정은 초기에 정의 필요.
* **DB 스키마는 개발 이전에 확정하는 방향이 효율적**
    * 실제 구현 과정 중 컬럼 추가/삭제가 반복되면서 테스트 시간 증가. ERD와 스키마는 우선 설계 후 진행이 안전함.
* **브랜치 전략 필요성 확인**
    * 브랜치 혼란은 충돌 처리 비용 증가를 초래. git-flow 기반 전략 적용이 적절함.
* **리마인더(스케줄러) 테스트는 시스템 시간 의존성 고려 필요**
    * 시간 기반 기능은 테스트 케이스를 명확히 준비해야 검증이 쉬움.

## 5. 결론

본 프로젝트는 다양한 리스크에 직면했지만, 문제 분석과 해결 과정을 반복하면서 비록 카카오 로그인 API를 포함한 프론트–백엔드 연동 기능을 마무리하지는 못하였지만, **인증 기능 안정화, 백엔드 서버 배포 성공, 이메일 자동화 기능 검증, Swagger API 테스트 환경 구축** 등을 완료하였다.

초기 예상 범위를 넘어서는 난이도였지만, 문제 해결 과정에서 다음 역량을 축적할 수 있었다.

1.  실서비스 환경 배포 경험
2.  외부 인증 연동 경험
3.  장애 대응 및 원인 분석 능력
4.  서버 로그 기반 디버깅 기술
5.  보안 설정 관리 능력

최종적으로 프로젝트 목표 기능을 완성하였으며, 기술적 안정성과 서비스 완성도를 모두 충족한 결과를 얻었다.

---

## 6. 프로젝트를 진행하며 느낀 점

이번 개인화 북마크 서비스 프로젝트는 저에게 정말 많은 걸 배우게 해 준 경험이었습니다. 특히 이번이 처음으로 진행해 본 서버 배포였기 때문에, 단순히 코드를 작성하는 수준을 넘어서 실제 서비스가 어떻게 세팅되고 운영되는지 직접 체감할 수 있었습니다. 로컬 환경에서는 잘 되던 게 배포 환경에서는 갑자기 안 되는 상황도 계속 발생했고, 서버 설정이나 DB 연결, 보안 같은 부분은 처음 겪는 내용이라 많이 헤맸지만 그 과정에서 확실히 시야가 넓어졌다고 느꼈습니다.

또 하나 기억에 남는 건 카카오 로그인 연동이었습니다. 외부 인증을 처음 연동해봐서 어렵게 느껴졌는데, Redirect URI나 토큰 발급 과정에서 오류가 계속 나서 시간이 많이 걸렸습니다. 그래도 직접 부딪히면서 OAuth 인증 구조를 이해할 수 있었고, 문제를 해결하려고 여러 자료를 찾아보고 테스트하면서 스스로 한 단계 성장한 느낌을 받았습니다.

재밌었던 건, 발표를 들었을 때 저랑 같은 오류를 겪었던 팀이 있었다는 점이었습니다. 그 팀은 결국 문제를 해결했고, 도와준다고 조언도 해줬는데, 저희 팀은 제출 기한 때문에 끝까지 해결을 못 했습니다. 강사님께서 “해결하면 가산점을 주겠다”고 하셔서 더 열심히 해봤는데 결국 못 받아서 아쉬웠습니다. 그래도 해결 방법을 고민하면서 방향을 제시해준 팀원들이 정말 고마웠고, 문제를 해결했던 민정이에게도 많은 도움을 받아 고마웠습니다. 지금 못했더라도 프로젝트가 끝나고 나서라도 꼭 해결해보고 싶은 마음이 생겼습니다.

전체적으로 이번 프로젝트는 단순한 수업 과제가 아니라, 진짜 서비스 개발 경험이었습니다. 로그 분석하는 법도 배우고, Swagger로 API 테스트하는 방법도 익히고, 이메일 리마인더 기능까지 구현하면서 개발 과정 전체를 경험했습니다. 오류가 날 때마다 답답하고 힘들었지만, 그걸 해결해 나가는 동안 개발자로서 자신감도 생겼습니다.

결과적으로 완벽한 마무리는 아니었지만, 앞으로 개발을 계속 해보고 싶다는 마음이 더 확실해졌고, 같은 문제를 다시 마주하더라도 이번보다 훨씬 잘 해결할 수 있을 것 같습니다. 무엇보다 같이 고민해주고 응원해준 팀원들 덕분에 끝까지 포기하지 않고 완성할 수 있었습니다. 이번 경험은 저에게 정말 의미 있었습니다.

좋은 경험할 수 있게 해주신 **차성원 강사님**께도 감사드립니다!
